# План по Spring

1. Введение в Spring Framework
   1. Spring
      1. Spring - это фреймворк уровня приложений, на нем можно построить все приложение
      2. Spring - индустриальный стандарт
      3. Spring состоит из нескольких подфремворков, проектов Spring, они решают какую-то задачу в приложении. Возможно использование нескольких подфремворков в одном приложении.
      4. Spring - "lightweight" фремворк
   2. Проекты Spring
      1. Spring IoC + AOP = Context
      2. Spring JDBC
      3. Spring ORM
      4. Spring Data, Spring Data JPA
      5. Spring MVC, Spring WebFlux
      6. Spring Security
      7. Spring Cloud
      8. Spring Boot
      
   3. Еще раз о проектах
      1. Spring Framework - предоставляет базовую функциональность: IoC, AOP, SpEL и т.д.
      2. Spring Boot - максимально ускоряеи разработку
      3. Spring Data - позволяет создавать JDBC-репозитории, написав только интефейс репозитория
      4. Spring Security - один из самых популярных фреймворков для реализации авторизации и аутентификации в приложениях
      5. Spring WebServices - позволяет максимально быстро создавать как сами веб-сервисы SOAP, так и клиентов к ним
      6. Spring MVC - позволяет создавать веб-приложения на основе MVC паттерна
      
   4. Spring IoC, Spring Context
      Базовая фукциональность.

      IoC - это название принципа объектного проектирования, согласно которому управление жизненным циклом некоего компонента происходит извне этого компонента. 
      DI - это частный случай, одна из возможных реализаций концепции Инверсии управления, применяемая, в частности, в Spring. Таким образом, с точки зрения Spring, DI и IoC - это одно и то же.
   
      Пример UserService :
         1. (-) UserService напрямую зависит от UserDAO
         2. (-) UserService невозможно тестировать отдельно от UserDAO
         3. (-) Жизненный цикл классов связан напрямую
         4. (-) Невозможно заменить UserDAO на другую реализацию
      
      Сильную связанность кода UserService можно уменьшить за счет паттерна ServiceLocator.
         1. (+) UserService не зависит от UserDAO, но зависит от ServiceLocator
         2. (+) Если постараться, то UserService можно тестировать отдельно от UserDAO
         3. (+) UserDAO можно заменить на другую реализацию
      
      Современный подход IoC-контейнер (Inversion of control): 
         1. (+) Контейнер создает необходимые объеты и управляет жизненным циклом
         2. (+) Класс UserService не зависит от UserDAO
         3. (+) UserService можно легко тетировать отдельно от UserDAO
         4. (+) UserDAO можно заменить на другую реализацию
      
      IoC - Inversion of Control (паттерн, принцип)
      DI - Dependency Injection
      
      IoC / DI
      1. (+) упрощает reuse компонентов
      2. (+) упрощает unit-тестирование
      3. (+) чистый код - только бизнес логика, никакого конфигурационного кода
      4. (+) хорошо вписывается в многослойную архитектуру (presentation layer/bisiness layer/data access layer)

      Зависимость от интерфейса наз Dependency Inversion.
      
      Как работает:
      1. читаются метаданные
      2. по метаданным читаются все необходимые классы
      3. создаются бины
      4. задаются зависимости

2. Конфигурирование Spring-Framework
   1. Java-based
      1. @Configuration
      2. @Bean

      1. Все бины располагаются в класса помеченных аннотацией @Configuration
      2. Классов @Configuration может быть много - обычно на каждый слой/технологию
      3. Бины создаются в нестатических методах, помеченных аннотацией @Bean
      4. Зависимости бинов - параметры методов
      
   2. Annotation
      1. @Component
      2. @Service
      3. @Repository
      4. @Controller
      
3. Расширенные методы конфигурирования Spring
   1. Bean scopes
      1. Singleton - создается один на все приложение, в зависимости подставляется один объект
      2. Prototype - объект не создается пока не является зависимостью другого бина, в каждый бин инжектится собственный экземпляр
      3. Request - бин создается на каждый web запрос и в каждом запросе он свой
      4. Session - то же, только в рамках Servlet HTTP-сессии
      5. Global session - то же, но только для портлетов (portlets)
   2. Beans lifecycle
      1. @PostConstruct
      2. @PreDestroy
   3. Profiles
   4. @Autowired
