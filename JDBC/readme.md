# План по JDBC
JDBC (Java Database Connectivity) - это стандартный API (набор классов и интерфейсов) на языке Java для взаимодействия с реляционными базами данных, входящий в состав JDK

1. Реляционная база данных
   1. Основные характеристики реляционных баз данных:
      1. Табличная структура: Данные хранятся в строках и столбцах, где каждая строка — это уникальная запись, а столбец — определенный тип атрибута.
      2. Связи (Relations): Таблицы могут быть связаны между собой, что позволяет избегать дублирования данных.
      3. Ключи: Первичный ключ (Primary Key) уникально идентифицирует строку в таблице, а внешний ключ (Foreign Key) обеспечивает связь с другой таблицей.
      4. ACID-согласованность: Реляционные БД обеспечивают атомарность, согласованность, изолированность и долговечность транзакций, что гарантирует надежность данных.
         1. Atomicity (Атомарность): Транзакция выполняется либо полностью, либо не выполняется совсем (если произошла ошибка, все изменения откатываются).
         2. Consistency (Согласованность): После завершения транзакции база данных переходит из одного корректного состояния в другое, соблюдая все правила и ограничения.
         3. Isolation (Изолированность): Параллельные транзакции не влияют друг на друга, промежуточные результаты одной транзакции не видны другим.
         4. Durability (Долговечность): Зафиксированные изменения (commit) сохраняются даже при сбое питания или системы.
      5. SQL (Structured Query Language): Стандартный язык для управления данными, позволяющий выполнять выборку, вставку, обновление и удаление информации.
   2. Типы данных MySQL
      1. Символьные типы
         1. CHAR: представляет строку фиксированной длины. (0-255)
         2. VARCHAR: представляет строку переменной длины. (0-65535)
      2. Числовые типы
         1. SMALLINT: представляет целые числа от -32768 до 32767, занимает 2 байт
         2. INT: представляет целые числа от -2147483648 до 2147483647, занимает 4 байта
         3. BIGINT: представляет целые числа от -9 223 372 036 854 775 808 до 9 223 372 036 854 775 807, занимает 8 байт
         4. DECIMAL: хранит числа с фиксированной точностью. Данный тип может принимать два параметра precision и scale: DECIMAL(precision, scale).
         5. FLOAT: хранит дробные числа с плавающей точкой одинарной точности от -3.4028 * 1038 до 3.4028 * 1038, занимает 4 байта. Может принимать форму FLOAT(M,D), где M - общее количество цифр, а D - количество цифр после запятой
         6. DOUBLE: хранит дробные числа с плавающей точкой двойной точности от -1.7976 * 10308 до 1.7976 * 10308, занимает 8 байт. Также может принимать форму DOUBLE(M,D), где M - общее количество цифр, а D - количество цифр после запятой
      3. Типы для работы с датой и временем
         1. DATE: хранит даты с 1 января 1000 года до 31 деабря 9999 года (c "1000-01-01" до "9999-12-31"). По умолчанию для хранения используется формат yyyy-mm-dd. Занимает 3 байта.
         2. TIME: хранит время от -838:59:59 до 838:59:59. По умолчанию для хранения времени применяется формат "hh:mm:ss". Занимает 3 байта.
         3. DATETIME: объединяет время и дату, диапазон дат и времени - с 1 января 1000 года по 31 декабря 9999 года (с "1000-01-01 00:00:00" до "9999-12-31 23:59:59"). Для хранения по умолчанию используется формат "yyyy-mm-dd hh:mm:ss". Занимает 8 байт
         4. TIMESTAMP: также хранит дату и время, но в другом диапазоне: от "1970-01-01 00:00:01" UTC до "2038-01-19 03:14:07" UTC. Занимает 4 байта
         5. YEAR: хранит год в виде 4 цифр. Диапазон доступных значений от 1901 до 2155. Занимает 1 байт.
      4. Составные типы
         1. ENUM: хранит одно значение из списка допустимых значений. Занимает 1-2 байта
         2. SET: может хранить несколько значений (до 64 значений) из некоторого списка допустимых значений. Занимает 1-8 байт.
      5. Бинарные типы
         1. BLOB: хранит бинарные данные в виде строки длиной до 65 КБ.
         LONGBLOB: хранит бинарные данные в виде строки длиной до 4 ГБ

4. Транзакции
   1. Транзакция в базе данных — это группа последовательных операций (запросов), объединенных в одну логическую единицу работы, которая выполняется по принципу «все или ничего». Она гарантирует, что данные либо полностью зафиксируются (commit) при успехе, либо полностью откатятся (rollback) при сбое, сохраняя целостность БД.
   2. Уровни изоляции (https://habr.com/ru/articles/845522/)
        1. READ UNCOMMITTED Это самый слабый уровень изоляции, когда транзакция может видеть результаты других транзакций, даже если они ещё не закоммичены.
        2. READ COMMITTED На этом уровне транзакция может читать только те изменения в других параллельных транзакциях, которые уже были закоммичены.
        3. REPEATABLE READ Этот уровень означает, что пока транзакция не завершится, никто параллельно не может изменять или удалять строки, которые транзакция уже прочитала.
        4. SERIALIZABLE Самый жёсткий, но самый тяжёлый для БД и медленный для обработки запросов уровень. Он блокирует любые действия, пока запущена транзакция — получается, транзакции идут строго одна за другой и максимально изолируются друг от друга.
